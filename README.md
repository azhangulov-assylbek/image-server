# Image Server — веб-приложение для хранения картинок

> **Учебный проект.** Эта работа — решение финального задания Модуля 2, выполненное
> в учебных целях под руководством ментора Тимерлана Чинчаева.
> Группа 10, курс Python Full Stack Developer, Python University Java Rush

Веб-приложение на Flask для загрузки, хранения и просмотра изображений. Файлы
сохраняются на диск, а их метаданные — в базе данных PostgreSQL. Приложение
упаковано в Docker и разворачивается одной командой через `docker-compose`.

## Возможности

- Загрузка изображений (перетаскиванием или выбором файла).
- Проверка файлов на сервере: разрешены только `jpg`, `png`, `gif`, размер до 5 МБ.
  Тип определяется по реальному содержимому файла (через Pillow), а не по расширению.
- Просмотр списка загруженных изображений с постраничной навигацией (по 10 на странице).
- Открытие и безопасное удаление изображений.
- Резервное копирование базы данных в SQL-дамп.
- Логирование событий в файл.

## Технологии

- **Python 3.12**, **Flask 3.0.3** — веб-приложение.
- **PostgreSQL 15** — хранение метаданных.
- **psycopg 3** — драйвер базы данных.
- **Pillow** — проверка и определение формата изображений.
- **nginx** — обратный прокси перед приложением.
- **Docker / docker-compose** — контейнеризация и запуск.

## Архитектура

Приложение состоит из трёх контейнеров, поднимаемых через `docker-compose`:

| Контейнер            | Образ                | Назначение                    | Порт (хост → контейнер) |
|----------------------|----------------------|-------------------------------|-------------------------|
| `image_server_nginx` | `nginx:1.27-alpine`  | Обратный прокси, точка входа  | `8080 → 80`             |
| `image_server_app`   | сборка из Dockerfile | Flask-приложение              | `3000 → 3000`           |
| `image_server_db`    | `postgres:15`        | База данных PostgreSQL        | `5435 → 5432`           |

Запрос проходит путь: **браузер → nginx (8080) → app (3000) → db (5432)**.

Данные сохраняются вне контейнеров:
- `./images` — загруженные файлы (примонтированы в `app` и `nginx`);
- `./logs` — лог-файл приложения;
- том `db_data` — данные PostgreSQL (переживают перезапуск контейнеров).

## Запуск

Нужен установленный **Docker** и **docker-compose**.

**1. Собрать и запустить контейнеры:**
```bash
docker-compose up --build
```

**2. Создать таблицу в базе (только при первом запуске).**
Таблица не создаётся автоматически — выполни один раз:
```bash
docker exec -it image_server_app python -m database.models
```
Должно вывести: `Таблица создалась успешно!`

**3. Открыть приложение в браузере:**
```
http://localhost:8080
```

Остановить: `docker-compose down` (данные сохранятся). Полностью удалить данные
базы: `docker-compose down -v`.

## Переменные окружения

Подключение к базе настраивается через переменные окружения (значения по умолчанию
заданы в `database/db.py` и подходят для работы внутри `docker-compose`):

| Переменная    | По умолчанию | Описание                          |
|---------------|--------------|-----------------------------------|
| `DB_HOST`     | `db`         | хост базы (имя сервиса в Docker)  |
| `DB_PORT`     | `5432`       | порт базы внутри сети Docker      |
| `DB_NAME`     | `images_db`  | имя базы данных                   |
| `DB_USER`     | `postgres`   | пользователь                      |
| `DB_PASSWORD` | `password`   | пароль                            |
| `IMAGES_DIR`  | `./images`   | папка для загруженных файлов      |
| `LOGS_DIR`    | `./logs`     | папка для логов                   |

> При локальном запуске `app.py` напрямую (без Docker) задай `DB_HOST=localhost`
> и `DB_PORT=5435`, чтобы достучаться до базы через проброшенный порт.

## Маршруты (эндпоинты)

| Метод и путь                     | Описание                                                |
|----------------------------------|---------------------------------------------------------|
| `GET /`                          | Главная страница.                                       |
| `GET /upload`                    | Страница загрузки изображений.                          |
| `POST /upload`                   | Загрузка файла (поле формы `image`). Возвращает JSON.   |
| `GET /images-list?page=N`        | Список изображений с пагинацией (10 на страницу).       |
| `GET /images/<filename>`         | Отдача файла изображения.                               |
| `POST /images/<filename>/delete` | Безопасное удаление изображения (файл + запись в БД).   |

Загрузка проверяет: наличие файла, непустоту, размер (≤ 5 МБ) и реальный формат
(`jpg`/`png`/`gif`). Файл сохраняется под уникальным именем (UUID), исходное имя
хранится в метаданных. Удаление защищено от выхода за пределы папки `images`
(path traversal) и работает только методом POST.

## База данных

Таблица `images`:

| Поле            | Тип                                      | Описание                        |
|-----------------|------------------------------------------|---------------------------------|
| `id`            | `SERIAL PRIMARY KEY`                     | идентификатор                   |
| `filename`      | `VARCHAR(100) UNIQUE`                    | имя файла на диске (UUID)       |
| `original_name` | `VARCHAR(100)`                          | исходное имя загруженного файла |
| `size`          | `INTEGER`                               | размер файла в байтах           |
| `upload_date`   | `TIMESTAMP` (по умолчанию текущее время) | дата и время загрузки           |
| `file_type`     | `VARCHAR(10)`                           | расширение файла                |

## Резервное копирование

Резервные копии базы создаются встроенной утилитой `pg_dump` и сохраняются в папку
`backups/` с датой и временем в имени (`backup_ГГГГ-ММ-ДД_ЧЧММСС.sql`).

**Автоматический бэкап (скрипт):**
```bash
python database/backup.py
```

**Ручной бэкап одной командой:**
```bash
docker exec -t image_server_db pg_dump -U postgres images_db > backup.sql
```

**Восстановление из дампа:**
```bash
cat backups/backup_2026-06-17_153000.sql | docker exec -i image_server_db psql -U postgres -d images_db
```

## Логирование

Приложение пишет события (успешные загрузки, ошибки, удаления) в файл
`logs/app.log` в формате `дата - имя - уровень - сообщение`.
